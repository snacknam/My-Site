사내 데이터를 활용하는 채팅 워크플로우

Exemble은 일반적인 LLM 채팅 서비스와 달리, 기업 내부의 데이터와 모델, 외부 API, 그리고 권한 체계를 함께 다뤄야 하는 제품입니다. 사용자는 일반적인 질문을 넘어, 사내 문서를 바탕으로 답변을 생성하거나 특정 워크플로우를 실행해 업무 결과물을 만들어 낼 수 있고, 관리자 입장에서는 이 과정을 운영하고 통제할 수 있어야 했습니다.

복잡한 AI 기능을 서비스 구조로 정리하기

사용자 화면은 AI를 실제로 사용하는 경험에 집중하고, 관리자 화면은 그 경험이 안정적으로 동작하도록 운영하는 구조에 집중하도록 방향을 정리했습니다. 같은 워크플로우, 데이터, 도구 메뉴라 하더라도 사용자는 ‘실행과 활용’ 관점에서, 관리자는 ‘설정과 통제’ 관점에서 접근하도록 설계했습니다. 이 과정에서 중요하게 봤던 것은 AI 기술을 많이 노출하는 것이 아니라, 사용자가 자신의 목적을 달성하기 위해 필요한 개념만 이해하도록 만드는 것이었습니다. 제품 내부는 복잡하지만 화면 위에서는 가능한 한 목적 중심으로 읽히도록, 정보구조와 메뉴 위계를 반복적으로 정리했습니다.

워크플로우를 제품의 핵심 경험으로 만들기

Exemble에서 가장 중요한 화면은 워크플로우였습니다. 단순한 설정 화면이 아니라, 데이터와 모델, 프롬프트, 도구를 연결해 실제 AI 실행 흐름을 설계하는 공간이었기 때문입니다. 사용자가 직접 워크플로우를 생성하고 수정하며 결과를 테스트할 수 있어야 했고, 동시에 복잡한 구조를 시각적으로 이해할 수 있어야 했습니다. 저는 이 화면을 단순한 노드 편집기가 아니라 업무 흐름을 설계하고 통제하는 인터페이스로 바라봤습니다. 그래서 노드 연결 자체보다도, 어떤 단계에서 어떤 역할이 수행되는지 이해할 수 있는 구조와 각 노드의 설정 맥락이 명확하게 드러나는 방식을 중요하게 다뤘습니다. 노드를 생성하고 연결하는 흐름, 세부 설정 패널, 실행 결과 확인까지 하나의 연속된 경험으로 연결되도록 설계했습니다. 또한 워크플로우는 기술적으로도 복잡한 화면이었기 때문에 구현 가능성과 사용성 사이의 균형이 중요했습니다. 시각적으로는 풍부하게 보이지만 실제로는 이해하기 어려운 방향보다는, 복잡성을 조금 덜어내더라도 빠르게 이해하고 수정할 수 있는 방향을 우선했습니다. 결과적으로 워크플로우는 Exemble의 핵심 기능이자, 제품 정체성을 가장 잘 보여주는 화면이 되었습니다.

LLM 채팅을 단순한 대화창이 아닌 실행 인터페이스로 설계하기

Exemble의 채팅 화면은 일반적인 AI 채팅 서비스와 닮아 보이지만, 실제 역할은 조금 달랐습니다. 사용자는 단순히 질문하고 답변을 받는 것뿐 아니라, 특정 워크플로우를 선택해 실행하거나 파일을 업로드하고 근거 자료를 확인하면서 결과를 검증해야 했습니다. 따라서 채팅 화면은 ‘대화 UI’인 동시에 ‘업무 실행 UI’여야 했습니다. 저는 이 화면에서 사용자의 흐름을 크게 두 가지로 나눠 보았습니다. 하나는 워크플로우를 선택하지 않고 기본 모델과 직접 대화하는 흐름이고, 다른 하나는 특정 워크플로우를 선택해 명확한 목적을 가진 실행을 수행하는 흐름입니다. 두 방식이 한 화면 안에서 자연스럽게 공존해야 했기 때문에, 워크플로우 선택 상태, 입력 경험, 결과 확인 구조를 분리하면서도 이질적으로 느껴지지 않게 조율했습니다. 또한 AI 응답 화면에서는 단순히 답변만 예쁘게 보여주는 것보다, 사용자가 지금 어떤 상태에 있는지 알 수 있도록 만드는 것이 중요했습니다. 입력 이후 스크롤 위치 변화, AI가 생각 중일 때의 애니메이션, 결과 생성 중 상태, 근거 자료를 우측 패널에서 확인하는 흐름, 파일 업로드와 같은 인터랙션을 세밀하게 다루었습니다. 이는 AI 제품에서 자주 발생하는 ‘무슨 일이 일어나고 있는지 모르겠다’는 불안을 줄이기 위한 설계이기도 했습니다.

데이터, 모델, 도구를 하나의 운영 경험으로 연결하기

Exemble은 채팅과 워크플로우만으로 완성되는 제품이 아니었습니다. 실제로 답변 품질과 실행 결과를 좌우하는 것은 그 뒤에 있는 데이터, 모델, 외부 도구의 관리 방식이었기 때문입니다. 사용자는 데이터 소스를 업로드하거나 연결해야 했고, 관리자는 어떤 모델을 사용할지 결정하고 필요한 API 도구를 등록해 워크플로우에서 활용할 수 있어야 했습니다. 이 영역에서 제가 중요하게 본 것은 기술 요소를 세세하게 드러내는 것보다, 사용자가 어떤 리소스를 준비하고 연결해야 원하는 결과를 얻을 수 있는지 명확하게 보여주는 것이었습니다. 데이터 메뉴에서는 업로드한 문서, 연결된 DB, 자동 수집 문서와 같은 데이터 소스를 관리하고, 벡터화 과정을 별도의 관리 맥락으로 분리해 구조를 이해하기 쉽게 만들었습니다. 모델 메뉴에서는 기본 정보와 리소스 정보, 파라미터, 학습 데이터를 통한 파인튜닝 흐름이 운영자의 관점에서 읽히도록 정리했습니다. 도구 메뉴 역시 단순한 API 등록 화면으로 보이지 않게 하려 했습니다. 실제로는 워크플로우 안에서 사용할 수 있는 기능 단위이기 때문에, 등록과 테스트, 연결 이후의 활용까지 자연스럽게 이어지는 경험이 되도록 설계했습니다. 이 과정에서 벡터화, 청킹, 임베딩, RAG 구조에 대한 이해는 단순한 기술 지식이 아니라, 사용자가 데이터를 어떻게 준비하고 AI 결과를 어떻게 신뢰할 수 있는지 설계하는 기반이 되었습니다.

권한과 인증키를 통해 엔터프라이즈 제품의 통제 구조를 설계하기

기업용 AI 제품에서 중요한 것은 단순히 기능이 많은가보다, 누가 어떤 데이터와 기능에 접근할 수 있는가를 명확하게 통제할 수 있는가였습니다. Exemble 역시 데이터, 모델, 워크플로우, 도구가 민감한 리소스이기 때문에, 권한 구조가 제품 전반에 일관되게 적용되어야 했습니다. 기존에는 메뉴 단위로 흩어진 권한 개념을 정리할 필요가 있었고, 최근에는 사용자/관리자/운영자 세 역할과 전체공개/비공개/초대 기반 공개 범위 개념을 바탕으로 권한 구조를 재편하고 있었습니다. 저는 이를 단순한 설정 항목이 아니라, 서비스 전반의 리소스 접근 모델로 바라보고 화면 구조에 반영했습니다. 계정 관리, 조직 관리, 벡터 테이블 접근 정책과 같이 서로 다른 관리 기능이지만, 결국은 “누가 무엇을 볼 수 있고, 사용할 수 있으며, 수정할 수 있는가”라는 하나의 질문으로 연결되도록 설계했습니다. 인증키 관리와 외부 도구 연결도 같은 맥락에서 중요했습니다. 외부 API를 등록하고 활용하는 경험은 편리해야 했지만, 동시에 민감한 정보 관리와 권한 통제도 분명해야 했기 때문입니다. 그래서 단순히 키를 발급하고 저장하는 기능이 아니라, 운영자가 신뢰하고 관리할 수 있는 화면이 되도록 구조와 표현을 조정했습니다.

대시보드와 로그로 운영 가능한 AI 제품을 만들기

AI 제품은 결과를 생성하는 화면만으로는 완성되지 않습니다. 특히 엔터프라이즈 환경에서는 제품이 잘 동작하는지, 어떤 문제가 발생했는지, 응답이 느려지지는 않는지, 자원이 안정적으로 사용되는지를 운영자가 계속 확인할 수 있어야 합니다. Exemble의 관리자 화면에서 대시보드와 로그는 바로 그런 운영 관점을 담당하는 핵심 영역이었습니다. 대시보드에서는 워크플로우 성공률, 요청 수, 응답 지연, 오류율과 같은 서비스 지표뿐 아니라 GPU 온도, VRAM 사용량, CPU/GPU 사용량, 메모리 등 리소스 상태를 함께 모니터링할 수 있도록 설계했습니다. 이때 중요한 것은 지표를 많이 보여주는 것이 아니라, 운영자가 서비스 상태를 빠르게 파악하고 이상 징후를 인식할 수 있는 구조를 만드는 것이었습니다. 로그 화면에서는 사용자 대화 내용, 워크플로우 응답, 참조한 근거 자료를 추적할 수 있도록 설계했습니다. 이는 단순한 이력 조회가 아니라, AI 결과를 신뢰하고 문제를 되짚어 볼 수 있는 기반이었습니다. 사용자 경험이 실제로 어떤 입력과 어떤 답변을 바탕으로 만들어졌는지 운영자가 확인할 수 있어야 했기 때문에, 로그와 근거 정보가 자연스럽게 이어지는 방식을 고민했습니다.

복잡한 AI 구조를 실제 제품 경험으로 바꾸는 역할

Exemble은 단순히 AI 기능을 예쁘게 보여주는 프로젝트가 아니었습니다. 사용자에게는 채팅과 워크플로우를 통해 실제 업무를 수행할 수 있는 경험을, 관리자에게는 데이터·모델·도구·권한·로그를 운영할 수 있는 경험을 함께 설계해야 하는 프로젝트였습니다. 저는 이 과정에서 기술 구조를 화면으로 옮기는 데 그치지 않고, 복잡한 요소들을 목적 중심의 제품 구조로 다시 정리하는 역할을 맡았습니다. 이 프로젝트를 통해 복잡한 시스템을 다룰수록 더 중요한 것은 기능의 양이 아니라, 사용자가 이해할 수 있는 구조와 운영 가능한 경험을 만드는 일이라는 점을 다시 확인했습니다. Exemble은 AI 기능을 담은 제품이 아니라, 복잡한 AI 시스템을 실제 기업 환경에서 사용할 수 있는 서비스로 바꾸는 작업이었습니다.
