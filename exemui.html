Exem UI  —  디자인 라이브러리로 사내 제품 통합하기

디자인 · 개발 통합 라이브러리 Exem UI · 2025.05 - 2026.02
Exem UI는 서로 다른 모습으로 존재하던 사내 제품들의 디자인을 하나의 공통 시스템으로 정리하고, 디자인과 개발이 같은 기준으로 함께 관리할 수 있도록 만든 통합 라이브러리입니다. 단순히 디자인 컴포넌트를 정리하는 것을 넘어 토큰 체계, 개발 라이브러리, 문서 사이트, 운영 정책까지, 지속 가능한 시스템을 주도적으로 구축하고 운영했습니다.

사내 제품 통합을 위한 공통 시스템

사내에는 여러 제품이 존재했는데, 각기 다른 시점에 각 팀의 방식대로 설계되면서 화면 구조, 디자인 스타일, 컴포넌트 사용 기준 등이 제품마다 다르게 형성되었습니다. 그 결과 제품의 사용자 경험이 마치 서로 다른 회사의 서비스처럼 일관되지 않았고, 제품이 확장될수록 화면 복잡도가 높아지면서 그에 따른 유지보수 비용이 점점 커지고 있었습니다.
서로 다른 모습의 제품 인터페이스

프론트엔드 환경에 기반한 라이브러리 구축

Exem UI는 디자인 파일 안에서만 끝나는 시스템이 아니라 실제 제품에 적용되는 라이브러리로서 기획됐습니다. 디자인과 개발 결과물이 서로 일치하는 구조를 만들기 위해 각 제품의 개발팀과 함께 구현 방식과 개발 환경을 조율했고, React와 Tailwind 등 실제로 사용될 기술 스택을 고려하여 디자인 시스템을 구축했습니다.
라이브러리 구현을 위한 개발 환경

디자인과 개발 구조의 정합성 설계

디자인 시스템을 구축하는 초기부터 컴포넌트 Props, 변수명, 상태 구조 등을 개발팀과 함께 논의해 실제 개발 구조와 최대한 일치하도록 설계했습니다. 이를 통해 디자인과 개발 간 해석 차이가 발생할 수 있는 여지를 줄이고, 컴포넌트 설계부터 구현까지 같은 기준으로 소통할 수 있는 기반을 마련했습니다.
디자인과 개발의 컴포넌트 구조 일치

디자인 컴포넌트 구조의 효율화

개발 환경뿐만 아니라 Figma 상에서도 디자이너가 컴포넌트를 효율적으로 사용하고 관리할 수 있도록 전체 구조를 모듈화했습니다. Nested Instance, Instance Swap, Slot 등 최신 Figma 기능을 활용해 하나의 컴포넌트가 다양한 맥락에서 사용될 수 있도록 설계했고, 이를 통해 더 적은 비용으로 일관된 화면을 빠르게 구성할 수 있는 환경을 만들었습니다.
다양한 형태로 모듈화된 컴포넌트 구조

컬러 토큰 기반의 다크 테마 체계

Exem UI의 컬러 시스템은 단순한 팔레트 정리를 넘어 의미 기반의 토큰 체계로 설계했고, Figma Variables를 활용해 라이트 · 다크 테마 모두에 대응할 수 있도록 구조를 정리했습니다. 특히 Opacity 기반의 Elevation 토큰을 만들어, 다양한 배경에서도 일관된 레이어 위계를 표현할 수 있는 시스템을 구현했습니다.
다크 테마와 Semantic 컬러 토큰 체계

모두가 기여할 수 있는 디자인 가이드

디자인 시스템은 특정 디자이너의 감각에 의존하면 오래 유지되기 어렵다고 판단하여, 누구나 같은 기준으로 작업하고 기여할 수 있는 디자인 가이드를 함께 정리했습니다. 이를 통해 신규 디자이너나 다른 제품 담당자도 명확하게 기준을 이해하고, 시스템 안에서 일관되게 작업할 수 있는 환경을 구축했습니다.
아이콘 디자인 가이드라인

사이트 배포를 통한 라이브러리 문서화

디자인 라이브러리의 확산과 활용성을 높이기 위해 Exem UI라는 별도의 사이트를 구축하고, 시스템의 구성 요소와 활용을 중심으로 전반적인 내용을 문서화했습니다. 이 사이트는 단순한 소개 페이지가 아니라 디자이너와 개발자가 동일한 기준을 참고하고, 변경 이력을 추적할 수 있는 운영 허브 역할을 하도록 설계되었습니다.
Exem UI 사이트

시스템 사용 맥락을 위한 Playground와 가이드라인

컴포넌트는 존재 자체보다 언제, 어떻게, 왜 써야 하는지가 더 중요하다고 판단해, 사이트 안에 Playground와 사용 가이드라인을 함께 정리했습니다. 이를 통해 팀원들은 단순히 UI 조각을 복붙하는 것이 아니라 각 컴포넌트의 적절한 사용 맥락과 예외 상황까지 이해하며 더 일관된 화면을 설계할 수 있었습니다.
Playground 화면 또는 컴포넌트 usage guideline 문서 이미지

지속 가능한 시스템 운영 정책 수립

라이브러리를 만든 사람 한 명에게 시스템이 과도하게 의존하면 장기적으로 유지되기 어렵기 때문에, 각 제품별 책임과 권한, 변경 요청 프로세스, 작업 주체를 구체적으로 정리했습니다. 그 결과 Exem UI는 단순한 산출물이 아니라 여러 팀이 함께 수정하고 확장할 수 있는 운영 체계를 갖춘 시스템으로 자리잡을 수 있었습니다.
운영 정책 다이어그램, 역할 분담표, 또는 요청 프로세스 플로우차트

AI 활용을 고려한 라이브러리 환경 구축

저는 Exem UI를 사람이 참고하는 문서에만 머무르게 하지 않고, AI나 LLM도 이해하고 활용할 수 있는 구조로 확장하고자 했습니다. CLI 패키지 설치 환경과 앞서 정리한 디자인 가이드라인을 연결해, 라이브러리의 규칙과 패턴이 생성형 도구 안에서도 일관되게 재사용될 수 있는 기반을 실험했습니다.
CLI 패키지 구조, AI 활용 흐름도, 또는 가이드 문서와 연결된 예시 이미지

Exem UI를 활용한 인터널 프로덕트 구축 사례

정리된 라이브러리와 가이드라인을 바탕으로 Claude Code를 활용해 사내 인터널 프로덕트를 빠르게 구축하며, 시스템이 실제 생성과 구현 단계에서도 유효한지 검증했습니다. 이 과정은 Exem UI가 단순한 정리 문서가 아니라 새로운 제품을 더 빠르고 일관되게 만드는 생산 기반이 될 수 있음을 보여주는 활용 사례였습니다.

디자인 시스템을 넘어 조직 공통의 기반 구축

Exem UI는 여러 제품의 시각적 일관성을 맞추는 작업을 넘어, 디자인과 개발이 같은 구조를 공유하고 조직이 지속적으로 사용할 수 있는 시스템을 구축한 프로젝트였습니다. 단순한 디자인 정리 넘어, 제품과 조직 전체의 기준을 설계하고 운영하는 플랫폼 디자이너의 역할을 구체적으로 확장할 수 있었습니다.
